## Scope

This block implements Origami's internationalization module. It includes locale file validation, compile-time `t()` key checking, interpolation variable verification, and the codegen that wires locale bundles into the generated Nuxt application.

Part of M3 (Feature-complete framework). The i18n module is opt-in: it activates when `"i18n"` is listed in `modules` in `origami.toml`. When inactive, `t()` is not a valid call and any use of it is a compile error.

---

## Expected output

- `t()` calls in `.ori` templates and logic sections validated at compile time
- Unknown keys, missing interpolation variables, and keys exceeding 3 segments are compile errors
- `origami-i18n`: validates locale files against the default locale, produces a `LocaleManifest`
- `origami-codegen` extended: emits processed locale bundles into `__generated__/locales/`
- Error codes CLT401–CLT403, W101–W103 implemented with `miette` messages
- Unit tests for key validation, interpolation checking, and locale file diffing

---

## Design principles

**Flat keys, not nested objects.** Locale files use flat string keys with a dot-separated convention, not nested JSON objects. Flat keys are easier to grep, LLM-friendly (no path navigation needed), and avoid structural duplication between locale files.

**Max 3 segments, enforced by the compiler.** The key structure `{context}.{component}.{label}` is a constraint, not a guideline. Keys with more than 3 segments are a compile error (CLT403). This keeps the key space navigable and prevents the slow sprawl that makes large i18n projects unmaintainable.

**Default locale as the canonical source.** The default locale file (`en.json`, or whatever `default_locale` is set to in `origami.toml`) is the only source of truth for which keys exist. Other locale files are validated against it: missing keys are warnings (W101), extra keys are warnings (W102). The build does not fail on missing translations — it fails on unknown keys in source code.

**`t()` is a compiler built-in.** It is available in all templates and logic sections without import. It does not exist at runtime as a library call — the compiler validates it statically and the codegen emits the appropriate Vue i18n call.

## Locale files

Locale files live in `locales/` at the workspace root, shared across all apps in the monorepo.

```
locales/
  ├── en.json     ← default locale, canonical key source
  ├── it.json
  └── fr.json
```

### Key naming convention

```
{context}.{component}.{label}
```

- `context`: page name or `common` for shared strings (`books`, `admin`, `common`)
- `component`: component or feature name, camelCase (`bookCard`, `filterForm`)
- `label`: specific string identifier (`title`, `saveButton`, `emptyState`)

Examples:

```json
{
  "common.loading":              "Loading...",
  "common.error":                "Something went wrong.",
  "common.save":                 "Save",
  "books.bookCard.title":        "Book",
  "books.bookCard.author":       "by {author}",
  "books.list.title":            "My Books",
  "books.list.empty":            "No books found.",
  "books.list.count":            "{count, plural, one {# book} other {# books}}"
}
```

Keys are flat strings. The 3-segment limit is enforced at the key string level — there is no JSON nesting. A key like `"admin.users.table.header.name"` (5 segments) is CLT403 regardless of its position in the file.

### Interpolation

`{variableName}` for substitution. `{variableName, plural, one {…} other {…}}` for plurals. Only `one` / `other` plural forms are supported in this version — complex ICU plural rules (e.g. `few`, `many`) are out of scope.

Date, time, and number formatting are delegated to platform locale APIs (`Intl`). The i18n module handles string keys and substitution only.

### Locale validation

The `origami-i18n` crate validates all non-default locale files against the default locale:

| Condition | Code | Severity |
|-----------|------|----------|
| Key in default locale, missing from other locale | W101 | Warning |
| Key in other locale, not in default locale | W102 | Warning |
| Key used in source but not in default locale | CLT401 | Error |

Warnings do not fail the build. Missing translations are expected during development; shipping a missing translation is a product decision, not a compiler error.

## Compiler pipeline extensions

### Lexer — no changes

`t()` uses the same token stream as any other function call. The lexer does not need to change.

### Parser — t() in templates

`t()` is treated as a special call expression in the template parser. The parser already handles function calls in prop values (e.g. `value={someFunc()}`). `t()` is recognized as a built-in by name during parsing and stored as a `TCall` node:

```
TCall
  ├── key: String            ← the key literal, e.g. "books.list.title"
  └── args: Option<TArgs>    ← interpolation variables, if any
```

`t()` is valid in:
- Template prop values: `<Text value={t('books.list.title')} />`
- Logic section expressions: `const label = t('common.save')`

`t()` is only valid when the i18n module is enabled. If the module is not listed in `origami.toml`, any `t()` call is a parse error.

### Analyzer — compile-time key validation

The analyzer validates every `TCall` node against the `LocaleManifest` produced by `origami-i18n`:

- CLT401: key string is not present in the default locale file
- CLT402: interpolation variable declared in the key's placeholder is not passed in the `t()` call
- W103: extra variable passed to `t()` with no matching placeholder in the key

The analyzer has read-only access to the `LocaleManifest` — it does not modify locale files.

## origami-i18n

New crate. Reads locale files, validates their structure and cross-locale consistency, and produces a `LocaleManifest` consumed by both the analyzer and `origami-codegen`.

### Input

```
locales/
  ├── en.json       ← parsed as default locale
  └── *.json        ← all other locales validated against default

origami.toml
  └── compiler.default_locale: String   ← "en" (default)
```

### Output

```
LocaleManifest
  ├── default_locale: String
  ├── keys: HashMap<String, LocaleKey>
  │         key → { value, placeholders: Vec<String> }
  └── locales: Vec<LocaleFile>
              ← all non-default locales with their keys
```

`placeholders` are extracted from the key value at parse time (e.g. `"by {author}"` → `["author"]`). The analyzer uses this to validate interpolation arguments in `t()` calls without re-parsing locale files.

## origami-codegen extension

`origami-codegen` receives the `LocaleManifest` and emits the processed locale bundles:

```
__generated__/
  └── locales/
      ├── en.json     ← processed default locale
      ├── it.json
      └── fr.json
```

The generated `nuxt.config.ts` is extended to register `@nuxtjs/i18n` with the processed locale files:

```typescript
// __generated__/nuxt.config.ts (extended)
export default defineNuxtConfig({
  modules: ['@nuxtjs/i18n'],
  i18n: {
    locales: ['en', 'it', 'fr'],
    defaultLocale: 'en',
    langDir: './locales/'
  }
})
```

`t()` calls in generated `.vue` files always use `useI18n()` destructured in `<script setup>`. The `t` function is then available in both the logic section and the template without any global reference:

```typescript
// in <script setup> — always emitted when i18n is active
const { t } = useI18n()

// logic section usage
const label = t('common.save')
```

```html
<!-- template usage — same t() from script setup, no global $t() -->
<Text :value="t('books.list.title')" />
```

The developer writes `t()` in `.ori` files — context-free, consistent in both template and logic sections. The codegen always emits the `useI18n()` destructure in `<script setup>`, regardless of whether `t()` appears in the template, the logic section, or both. `$t()` is never emitted — it is a legacy global that bypasses the Composition API and creates implicit dependencies.

## Runtime locale switching

`origami dev` loads all locale files by default. The `--locale` flag restricts loading to a single locale (useful for focused work on a specific language):

```bash
origami dev                  # loads all locales, runtime switching available
origami dev --locale it      # loads Italian only (faster startup)
origami build                # bundles all locales
origami build --locale it    # bundles Italian only (single-market deploy)
```

Runtime locale switching (changing the active language at runtime without a page reload) is handled by `@nuxtjs/i18n` — it is not something Origami implements. The compiler's job is to validate keys and emit correct bundles; the runtime switching mechanism belongs to the i18n library.

## Error codes

| Code | Condition | Phase |
|------|-----------|-------|
| CLT401 | `t()` key not found in default locale file | analyzer |
| CLT402 | Interpolation variable missing from `t()` call | analyzer |
| CLT403 | Locale key exceeds 3 segments | i18n validation |
| W101 | Key missing from a non-default locale file | i18n validation |
| W102 | Extra key in a non-default locale file | i18n validation |
| W103 | Extra interpolation variable passed to `t()` | analyzer |

CLT401 and CLT402 are errors — unknown keys and missing variables are always bugs. W101 and W102 are warnings — incomplete translations are a product concern, not a compiler concern.

## CLI contribution

With this block, `origami check` additionally validates:
- All locale files parse as valid JSON
- No key in any locale file exceeds 3 segments (CLT403)
- All non-default locales are diffed against the default (W101, W102)

`origami build` is extended to invoke `origami-i18n` and write `__generated__/locales/`.

`origami dev` and `origami build` gain the `--locale` flag.

## Tests

**Unit tests in `origami-i18n/src/tests.rs`:**
- Key parsing: segments count, placeholder extraction
- Cross-locale diffing: missing keys (W101), extra keys (W102)
- CLT403: key with 4+ segments
- Placeholder extraction from plural forms

**Unit tests in `origami-analyzer/src/tests.rs` (extended):**
- CLT401: `t()` call with unknown key
- CLT402: `t()` call missing a required interpolation variable
- W103: `t()` call with extra variable

```
fixtures/
```
